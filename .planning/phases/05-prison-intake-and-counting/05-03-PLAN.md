    **requirePrisonSession():**
    ```typescript
    import { auth } from '@/auth'
    import type { JWTClaims } from '@repo/db' // or wherever the type lives

    async function requirePrisonSession(): Promise<JWTClaims> {
      const session = await auth()
      if (!session?.user || session.user.role !== 'prison') {
        throw new Error('Unauthorized: prison role required')
      }
      return {
        ...session.user,
        sub: session.user.id!,
      } as JWTClaims
    }
    ```
    Match the exact pattern from Phase 4's `requireRecoAdmin` but checking for `prison` role. The `facility_id` is already on `session.user` from the Custom Access Token Hook (Phase 2). Use `session.user.id` mapped to `sub` — same as Phase 4 decision (page.tsx uses auth() directly).

    **getExpectedDeliveries():**
    Exported Server Action. Queries:
    1. Pickups with status `delivered` that have a transport_booking with `destination_prison_id` matching the user's facility_id, AND no existing intake_record for that pickup_id.
    2. For each pickup: join location (for origin market/country), pickup_lines (for product quantities), tenant (for client name).
    3. Group results: direct deliveries (transport_type = 'direct') as individual items, consolidated deliveries grouped by outbound_shipment_id (via outbound_shipment_pickups).

    Return shape:
    ```typescript
    type ExpectedDelivery = {
      pickup_id: string
      reference: string
      client_name: string
      tenant_id: string
      origin_market: string
      expected_date: string
      transport_type: 'direct' | 'consolidation'
      outbound_shipment_id?: string
      outbound_shipment_reference?: string
      lines: { product_id: string; product_name: string; quantity: number }[]
    }

    type DeliveryGroup = {
      type: 'direct'
      delivery: ExpectedDelivery
    } | {
      type: 'consolidated'
      outbound_shipment_id: string
      outbound_shipment_reference: string
      deliveries: ExpectedDelivery[]
    }
    ```

    Use `withRLSContext(user, ...)` for all queries — prison_role RLS on pickups/transport tables enforces facility isolation. The query logic:
    - SELECT from pickups WHERE status = 'delivered'
    - JOIN transport_bookings ON transport_bookings.pickup_id = pickups.id
    - LEFT JOIN outbound_shipment_pickups for consolidation grouping
    - LEFT JOIN outbound_shipments for shipment reference
    - LEFT JOIN intake_records ON intake_records.pickup_id = pickups.id to EXCLUDE already registered
    - WHERE intake_records.id IS NULL (only unregistered deliveries)
    - JOIN locations ON pickups.location_id = locations.id (for country/city)
    - JOIN tenants ON pickups.tenant_id = tenants.id (for client name)

    Also fetch pickup_lines with product names for each pickup in a second query (or use a subquery/lateral join).

    Group the results in application code: direct deliveries as `{ type: 'direct', delivery }`, consolidated as `{ type: 'consolidated', outbound_shipment_id, outbound_shipment_reference, deliveries: [...] }`.
