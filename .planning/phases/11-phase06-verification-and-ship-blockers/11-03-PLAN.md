# Phase 11 — Plan 03: Local Beta Test Checklist (Full Lifecycle E2E Smoke Test)

## Goal

Provide a complete, module-by-module manual test checklist that a developer can execute against a locally running instance to validate all major user flows before the beta goes live.

---

## Context and Constraints

- **No SMTP configured for beta.** Email delivery verification means confirming that a notification record was created in the `notifications` table and that the notification centre bell shows the correct unread count. It does NOT mean checking a real inbox.
- **Magic link in dev mode.** Auth.js v5 in development mode logs the magic link URL to the terminal console. No real email needs to be delivered to test the auth flow.
- **Microsoft Entra ID is not testable locally** without a real Azure app registration. Test only magic link auth during beta smoke testing.
- **ESG-04 CO2 column is intentionally null.** This is an acknowledged formula stub — it should display "Formula pending" or similar placeholder, not an error.
- **Photo uploads** (intake photo attachments) require `SUPABASE_URL` and `SUPABASE_ANON_KEY`. If these are not configured, skip photo upload steps and note the gap.
- **Module-by-module testing is acceptable.** You do not need to run all modules in a single browser session.

---

## Environment Setup

Complete this section before running any module tests.

### Step 1 — Configure environment

- [ ] Copy `.env.example` to `apps/web/.env.local`
- [ ] Fill in `DATABASE_URL` pointing to a local PostgreSQL instance or Supabase free-tier cloud project
- [ ] Generate `AUTH_SECRET`: run `npx auth secret` and paste the output
- [ ] Set `AUTH_RESEND_KEY=re_test_xxx` (magic link URL will be logged to the dev console — no real email needed)
- [ ] Set `RESEND_API_KEY=re_test_xxx` (server will not crash; emails will silently not deliver)
- [ ] Set `NEXT_PUBLIC_DOMAIN_MODE=azure-default`
- [ ] Set `NEXT_PUBLIC_APP_DOMAIN=localhost:3000`
- [ ] Optionally set `SUPABASE_URL` and `SUPABASE_ANON_KEY` if photo upload testing is desired

### Step 2 — Build the database

- [ ] Run `bash scripts/setup-local-db.sh`
- [ ] Confirm the script exits 0 with no errors
- [ ] Spot-check: `psql $DATABASE_URL -c "\dt"` should list 20+ tables including `pickups`, `intake_records`, `processing_reports`, `outbound_dispatches`, `audit_log`, `notifications`, `import_jobs`
- [ ] Spot-check: `psql $DATABASE_URL -c "SELECT count(*) FROM tenants"` should return > 0 (Wolt seed)

### Step 3 — Build verification

- [ ] Run `pnpm --filter @apps/web tsc --noEmit` — must exit 0 (or document pre-existing errors that are not Phase 06 related)
- [ ] Run `pnpm turbo build` — must exit 0 with no build errors
- [ ] Confirm dev server starts: `pnpm --filter @apps/web dev` — visit `http://localhost:3000`

## Module Test Checklists

### Module A — Tenant and Auth Setup

Goal: Confirm the app resolves correctly and magic link login works.

- [ ] Visit `http://localhost:3000` — page loads without a 500 error
- [ ] The page resolves to the correct context (public or ops login screen)
- [ ] Click "Sign in with magic link" (or equivalent), enter a test email address
- [ ] Observe the dev server terminal — the magic link URL is printed to console (look for a line containing `/api/auth/callback?token=...`)
- [ ] Open the magic link URL in the browser — session is created and user is redirected to the ops dashboard
- [ ] Confirm the logged-in user's role can access `/admin` if they have the `reco-admin` role (if seed data includes a reco-admin user, use that email)
- [ ] Confirm a prison-staff user cannot access `/admin` — they should receive a 403 or be redirected

### Module B — Pickup Booking

Goal: Confirm client users can submit pickup requests and the ops queue reflects them.

- [ ] Log in as a client-role user (or assign client role to test user in DB)
- [ ] Navigate to the pickup booking form (route: `/pickups/new` or equivalent)
- [ ] Fill in all required fields and submit
- [ ] Confirm a `PU-YYYY-NNNN` reference number is generated and displayed on the confirmation screen or in the pickup list
- [ ] Log in as reco-admin and navigate to the ops pickup queue
- [ ] Confirm the new pickup appears with status `pending` (or first status in lifecycle)
- [ ] Confirm the PU reference number matches what the client saw

### Module C — Transport Booking and Prison Intake

Goal: Confirm the transport and intake flow from reco-admin booking through to prison staff recording the intake.

- [ ] Log in as reco-admin
- [ ] Navigate to a confirmed pickup record and book a direct transport (fill transport form, submit)
- [ ] Log in as a prison-staff user
- [ ] Navigate to the prison tablet view — confirm the expected delivery appears in the incoming deliveries list
- [ ] Click through to the intake form and fill in bag count / quantity
- [ ] Submit the intake form — confirm an `IN-YYYY-NNNN` reference number is generated
- [ ] **Discrepancy test:** Enter a quantity that is more than 20% above the informed quantity (e.g. if informed = 100, enter 121)
- [ ] After submitting, run: `psql $DATABASE_URL -c "SELECT discrepancy_flagged FROM intake_records ORDER BY created_at DESC LIMIT 1"` — should return `t` (true)
- [ ] Confirm the ops queue shows the intake with a discrepancy flag indicator

### Module D — Processing and Dispatch (Phase 06 Focus)

Goal: Confirm the full processing pipeline and dispatch flow work end-to-end.

**Processing form — max 2 taps:**

- [ ] Log in as prison-staff
- [ ] From the prison home screen, navigate to the processing form — count the number of taps/clicks required (must be ≤2 from the home screen)
- [ ] Select activity type "Wash" and fill in all fields: staff name, client, product, date, notes
- [ ] Submit the wash report — confirm success (no error, record appears in pipeline)
- [ ] Create a second report with activity type "Pack" — fill in size bucket quantities (XS, S, M, L, XL or equivalent)
- [ ] Submit the pack report — confirm success

**Pipeline view:**

- [ ] Navigate to the processing pipeline view (route: `/processing` or `/prison/processing`)
- [ ] Confirm the pipeline shows at least the stages: awaiting processing / in progress / ready to ship / shipped
- [ ] Confirm the wash and pack records appear in the correct stage

**Dispatch:**

- [ ] Log in as reco-admin
- [ ] Navigate to the dispatch creation form
- [ ] Create a new outbound dispatch record for the processed items
- [ ] Add at least one packing list line item (product, quantity)
- [ ] Submit — confirm dispatch is created with status `created`
- [ ] Update dispatch status to `picked_up` — confirm transition succeeds
- [ ] Update dispatch status to `delivered` — confirm transition succeeds
- [ ] Attempt to update a `delivered` dispatch to `created` — confirm this is rejected (invalid transition)

**Prison dispatch read-only view:**

- [ ] Log in as prison-staff
- [ ] Navigate to `/prison/dispatch` (or equivalent dispatch history route)
- [ ] Confirm the dispatch created above appears in the list
- [ ] Confirm there are no "Create", "Edit", or "Delete" buttons visible — the view is read-only for prison staff

**Traceability chain:**

- [ ] Navigate to the detail view of the intake record created in Module C
- [ ] Confirm a traceability chain is rendered showing: Pickup → Transport → Intake → Processing (Wash) → Processing (Pack) → Dispatch
- [ ] Each link should be a navigable reference or at minimum display the reference number

### Module E — Audit Trail

Goal: Confirm edit locking, admin bypass, void, and trigger-based audit log all work.

- [ ] Log in as prison-staff
- [ ] Edit an intake record that was created within the last 48 hours — edit should succeed
- [ ] After editing, reload the intake list — confirm the "Edited" indicator (pencil icon, badge, or similar) is visible on the record
- [ ] Click the edited indicator — confirm it links to or displays the edit history for that record

**48-hour lock test:**

- [ ] Identify an intake record's `created_at` timestamp in the DB
- [ ] Run: `psql $DATABASE_URL -c "UPDATE intake_records SET created_at = now() - interval '49 hours' WHERE id = '<record-id>'"` to backdated the record
- [ ] As prison-staff, attempt to edit that record — confirm the edit is rejected with a "record is locked" or equivalent error message
- [ ] As reco-admin, attempt to edit the same backdated record — confirm the edit succeeds (admin bypasses the lock)

**Void:**

- [ ] As reco-admin, void an intake record using the void action (VoidRecordDialog or equivalent)
- [ ] Confirm the record no longer appears in the standard intake list (voided records are excluded)
- [ ] Run: `psql $DATABASE_URL -c "SELECT voided FROM intake_records ORDER BY updated_at DESC LIMIT 1"` — should return `t`
- [ ] Navigate to the audit trail / void history view — confirm the voided record is visible there

**Trigger verification:**

- [ ] After completing any edit above, run: `psql $DATABASE_URL -c "SELECT * FROM audit_log ORDER BY created_at DESC LIMIT 5"`
- [ ] Confirm rows exist in `audit_log` — the trigger should have fired on the UPDATE
- [ ] If `audit_log` is empty, this is a **ship blocker**: `0005_phase6_processing_dispatch_audit.sql` was not applied. Run it manually: `psql $DATABASE_URL -f packages/db/migrations/0005_phase6_processing_dispatch_audit.sql` and re-test

### Module F — Financial Tracking

Goal: Confirm financial records are auto-created and are editable by reco-admin.

- [ ] Navigate to `/financial` (or equivalent financial records route) as reco-admin
- [ ] Confirm that an intake record with status `delivered` has an auto-created financial record
- [ ] Edit the invoice status (e.g. from `draft` to `invoiced`) — confirm the change saves
- [ ] Confirm that the EUR/DKK currency toggle works (switching currency recalculates displayed amounts)
- [ ] Confirm imported records (from Module I, if run) show the blue "Imported" badge in the financial list

### Module G — Dashboards and ESG Metrics

Goal: Confirm dashboards load with real data and ESG export works.

- [ ] Log in as reco-admin
- [ ] Navigate to the ops dashboard — confirm it loads without errors and shows summary cards for pickups and intakes
- [ ] Navigate to the ESG metrics page
- [ ] Confirm material weight totals are shown (not zero, not an error — should reflect seed or test data)
- [ ] Confirm the CO2 column/card shows a "Formula pending" placeholder or equivalent (not an error, not a stack trace)
- [ ] Click the CSV export button — confirm a `.csv` file downloads
- [ ] Open the downloaded CSV — confirm it contains rows with intake data and that column headers are present

### Module H — Notifications (No SMTP)

Goal: Confirm notification records are created in the DB and appear in the notification centre. Real email delivery is not required.

- [ ] Trigger a discrepancy: repeat the Module C discrepancy test (intake quantity > 20% above informed)
- [ ] Run: `psql $DATABASE_URL -c "SELECT * FROM notifications ORDER BY created_at DESC LIMIT 3"` — confirm a new notification row was created with a type like `discrepancy_detected` or equivalent
- [ ] Navigate to the notification centre bell icon in the ops nav
- [ ] Confirm the bell shows an unread count (badge number > 0)
- [ ] Click the notification — confirm it displays the discrepancy message and links to the intake record
- [ ] Mark the notification as read — confirm the unread count decreases
- [ ] **Email non-crash verification:** Check the dev server terminal logs — there should be no uncaught exception or 500 error from the Resend SDK when `RESEND_API_KEY=re_test_xxx`. A silently failed send (Resend returns 4xx which is caught and logged) is acceptable.

### Module I — Historical Import

Goal: Confirm the import wizard works end-to-end for at least one source.

Prepare a small test CSV file with 5 rows matching the `intake_log` source schema. Required columns (at minimum): `facility_name`, `staff_name`, `delivery_date`, `product_name`, `quantity`. Include one intentionally bad row (e.g. missing `delivery_date`) to test validation.

- [ ] Log in as reco-admin
- [ ] Navigate to `/import` — confirm 5 source cards are visible
- [ ] Click "Prison Intake Log" — confirm the import wizard opens at step 1 (file upload)
- [ ] Upload the test CSV — confirm the wizard advances to the column mapping step
- [ ] Map at least one column manually (drag or select) — confirm the mapping is reflected in the preview
- [ ] Advance to the validation preview step — confirm the bad row is highlighted with a field-level error message
- [ ] Advance to commit — confirm only valid rows are committed (bad row skipped or flagged)
- [ ] After commit, navigate to the intake list — confirm the committed records show a blue "Imported" badge
- [ ] Attempt to navigate back to the same import job URL and click Commit again — confirm the error "Job already committed" is shown

## Known Acceptable Gaps

Document these gaps in the beta test report. They are not blockers for the beta, but must be resolved before go-live.

| Gap | Severity | Resolution |
|-----|----------|------------|
| Microsoft Entra ID (Azure SSO) not testable without a real Azure app registration | Medium | Create Azure app registration before go-live; not required for beta |
| Magic link email delivery not verified (console log only) | Low | Real Resend key required for production; dev mode console log is sufficient for beta |
| ESG-04 CO2 column intentionally null | Low | Acknowledged formula stub; populate before go-live |
| Photo upload to Supabase Storage untested if `SUPABASE_URL` not configured | Medium | Configure Supabase credentials to test; does not block other modules |
| `audit_log` trigger requires manual `psql` application of `0005_phase6_processing_dispatch_audit.sql` to any live DB | High | Include in production deployment runbook; covered by `setup-local-db.sh` locally |
